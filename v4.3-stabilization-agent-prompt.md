# LeeDeli v4.3 안정화 실행 프롬프트

## 너의 역할
너는 LeeDeli 프로젝트에 처음 투입된 코딩 에이전트다.
목표는 기능 추가가 아니라 안정화다.
이번 작업은 v4.3 보고서의 미완료 항목 중 P1 범위를 닫는 것이다.

## 프로젝트 한 줄 설명
카페 운영 도구이며 React 19 + TypeScript + Supabase + Playwright를 사용한다.

## 이번 작업의 최종 목표
아래 4개 게이트를 모두 통과하면 완료다.
1. auth.spec의 unauthorized 시나리오가 연속 통과한다.
2. storage.spec의 기존 flaky 케이스가 반복 실행에서 안정 통과한다.
3. typecheck와 build가 모두 성공한다.
4. v4.3 기준 P1 미해결 항목을 해결 완료로 기록한다.

## 완료 범위와 제외 범위
포함 범위:
- Auth 상태 전이 안정화
- Mock E2E flaky 완화
- build/typecheck/핵심 mock 테스트 통과
- migration 004, 005 정합성 검증

제외 범위:
- preps 스키마 장기 통일
- src/v2 제거
- smoke E2E 대규모 확장
- UI 리디자인

## 시작 전에 반드시 확인할 사실
1. migration 004와 005 파일은 이미 저장소에 존재한다. 생성 작업이 아니라 검증 작업이다.
2. AuthContext에는 INITIAL_SESSION 무시 로직이 이미 존재한다. 먼저 현재 실패가 실제로 재현되는지 확인해야 한다.
3. flaky의 핵심 원인은 networkidle 의존 가능성이 높다.

## 작업 규칙
1. Root cause를 먼저 확인하고 수정한다. timeout 확대만으로 닫지 않는다.
2. 한 번에 많은 변경을 넣지 말고 단계별로 검증한다.
3. 테스트는 단발 통과가 아니라 반복 통과로 안정성을 판단한다.
4. 보고서와 실제 코드가 다르면 실제 코드를 기준으로 결정하고, 차이를 기록한다.
5. 필요 시 harness 재실행을 위해 feature_list 상태를 failing으로 내린다.

## 작업 순서

### Phase 0. 기준선 고정
목적: 무엇이 진짜 문제인지 먼저 확정한다.

실행:
1. 아래 파일을 읽고 현재 구조를 파악한다.
- src/auth/AuthContext.tsx
- src/components/RequireAuth.tsx
- src/auth/RequireAdmin.tsx
- e2e/mock/auth.spec.ts
- e2e/mock/auth-flow.spec.ts
- e2e/mock/storage.spec.ts
- playwright.config.ts
- src/storage/prepsRepo.ts
- supabase/migrations/004_fix_admin_rls_recursion.sql
- supabase/migrations/005_preps_add_data_column.sql

2. 기준선 테스트를 실행한다.
- npm run typecheck
- npm run build
- npm run test:mock -- e2e/mock/auth.spec.ts
- npm run test:mock -- e2e/mock/storage.spec.ts

3. 재현 결과를 3축으로 기록한다.
- auth 일관 실패 여부
- storage flaky 여부
- build/typecheck 통과 여부

산출물:
- 기준선 결과 요약
- v4.3 문서와 현재 코드의 차이 목록

### Phase 1. Auth 근본수정
진입 조건: auth 이슈가 재현되거나 간헐 실패가 확인된 경우

핵심 의도:
- session 판단과 admin 판단의 준비 상태를 분리해 타이밍 경합을 제거한다.

작업:
1. AuthContext의 로딩 모델을 분리한다.
- 단일 loading에 session 완료와 admin 완료를 섞지 않는다.
- session 준비와 admin 준비를 각각 관리한다.

2. 게이트 컴포넌트를 목적별로 분리한다.
- RequireAuth는 session 준비 상태만 본다.
- RequireAdmin은 admin 준비 상태만 본다.

3. auth.spec 대기 전략을 화면 조건 중심으로 조정한다.
- networkidle에만 의존하지 않는다.
- 실제 사용자 관찰 조건(heading, 버튼 visible)을 기준으로 검증한다.

4. 재현성 검증
- npm run test:mock -- e2e/mock/auth.spec.ts 연속 2회
- npm run test:mock -- e2e/mock/auth-flow.spec.ts 1회

완료 기준:
- unauthorized 시나리오가 연속 통과한다.
- 로그인 리다이렉트와 redirect-back 흐름이 깨지지 않는다.

### Phase 2. Mock E2E flaky 안정화
진입 조건: storage 케이스에서 flaky가 관찰된 경우

작업:
1. storage.spec에서 networkidle 의존 지점을 찾는다.
2. 페이지별 핵심 DOM 조건 대기로 교체한다.
3. playwright 설정에서 안정화 파라미터를 조정한다.
- workers=1 유지
- 필요 시 webServer timeout 상향
- 필요 시 navigation/action timeout 명시

검증:
- npm run test:mock -- e2e/mock/storage.spec.ts 연속 3회

완료 기준:
- 3회 중 3회 통과
- 첫 실행과 재실행 간 편차가 줄어든다

### Phase 3. 정합성 및 품질 게이트
작업:
1. migration 004, 005 정합성을 검증한다.
- 파일 존재
- SQL 멱등성 패턴 사용
- prepsRepo 기대 스키마와 일치

2. 최종 품질 게이트 실행
- npm run typecheck
- npm run build
- npm run test:mock -- e2e/mock/auth.spec.ts
- npm run test:mock -- e2e/mock/storage.spec.ts

3. 필요 시 harness 재평가 큐를 설정한다.
- feature_list.json에서 S3T2, S4T1, S4T2를 failing으로 조정

완료 기준:
- 위 4개 명령이 모두 통과
- 하네스 재실행 조건이 명확히 기록됨

## 실패 시 디버깅 지침
1. 먼저 실패를 다시 재현한다.
2. 실패 로그에서 최초 실패 지점을 확정한다.
3. 관련 컴포넌트 상태 전이를 추적한다.
4. 가설 하나만 세워 최소 변경으로 검증한다.
5. 통과 후 회귀 테스트를 실행한다.
